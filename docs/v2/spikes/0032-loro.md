# Issue 32: Loro native/WASM convergence spike

Status: executable spike; protocol decisions remain subject to issue #33

## Decision under test

ResearchPocket uses the same Rust domain crate in native clients and the hosted
browser. The crate pins `loro = "=1.13.6"` and compiles to both an `rlib` and a
`cdylib`. It does not ship the separate `loro-crdt` npm runtime, so there is one
implementation of validation, mutation, convergence, and projection semantics.

Loro resolves application conflicts. Git may store immutable update envelopes,
but commit order, merge results, timestamps, and branch topology never choose
user-visible state.

## Spike schema

Dynamic nested containers use Loro 1.13's `ensure_mergeable_*` APIs. Regular
operation-identified child containers are deliberately excluded because two
offline replicas can otherwise create different children at the same map key.

```text
items/<item-id>/
  note: LoroText
  scalars/title/revisions/<revision-id>: immutable canonical JSON
  tags/<normalized-tag>/adds/<add-dot>: true
  tags/<normalized-tag>/removes/<observed-add-dot>: true
  lifecycle/revisions/<revision-id>: immutable canonical JSON
```

- Notes use explicit UTF-16 offsets in every runtime. Rust's default Loro text
  offsets are Unicode scalar positions, while browser editors use UTF-16.
- Scalar revisions form a causal graph. Every edit retains the observed heads;
  concurrent heads remain recoverable and a revision-ID order selects the
  deterministic visible value.
- Tags are add-wins observed-remove sets. A removal records only add dots that it
  observed, so an unseen concurrent add survives.
- Lifecycle revisions carry explicit generations and observed parents. Deletion
  wins when any maximal head is deleted. Restore becomes visible only after it
  observes every concurrent delete head.

The canonical projection is an explicit Rust struct allowlisting only title,
note, tags, lifecycle, and their required revision evidence. Raw Loro state,
transport metadata, unknown map keys, and wall-clock values are excluded.

## Immutable envelopes

Each fixture mutation exports `ExportMode::updates(&previous_version)` into a
versioned envelope. Loro peer IDs and the durable sequence use strings in JSON to
avoid JavaScript's integer limit. The SHA-256 digest covers the decoded update.
The spike validates both the external digest and Loro's embedded checksum before
import.

An identical envelope can be replayed repeatedly. Updates can arrive before
their causal dependencies; Loro holds them pending and applies them when the
missing immutable envelopes arrive. The shared scenario replays the same set in
ordered, reversed, and duplicated schedules and requires byte-for-byte identical
canonical JSON.

The durable ResearchPocket device UUID is not a Loro peer ID. Production uses a
fresh Loro peer ID for each concurrent process or browser tab unless a future
design supplies strict single-writer locking and restores the complete local
peer history first.

## Verification

```sh
cargo test -p research-domain
cargo check -p research-domain --target wasm32-unknown-unknown
wasm-pack test crates/research-domain --headless --chrome
wasm-pack build crates/research-domain --target web --release
```

Native and browser tests invoke the same `run_convergence_scenario` function.
The fixture covers UTF-16 concurrent note insertion, deletion and replacement;
retained scalar conflicts; add-wins tags; partial and complete lifecycle
restores; and reordered/duplicated update envelopes.

Verified on 2026-07-10 with Rust 1.97.0 and Chrome/ChromeDriver 150.0.7871.115:

- `cargo fmt --package research-domain -- --check`: passed;
- `cargo clippy -p research-domain --all-targets -- -D warnings`: passed;
- `cargo test -p research-domain --all-targets`: passed;
- `cargo check -p research-domain --target wasm32-unknown-unknown`: passed;
- the browser-WASM test, run through `wasm-bindgen-test-runner` 0.2.126 in
  headless Chrome: one passed, zero failed; and
- `cargo build -p research-domain --target wasm32-unknown-unknown --release`:
  passed.

After `wasm-bindgen 0.2.126 --target web`, the release WASM measures 3,667,505
bytes raw, 1,027,929 bytes with gzip level 9, and 693,619 bytes with Brotli level
11. These are spike measurements, not a final budget: the workspace does not yet
apply a size-oriented release profile or `wasm-opt`.

## Constraints and follow-up

- Checkpoints must use full `ExportMode::Snapshot` while V2 retains complete
  history. Shallow snapshots cannot accept operations older than their frontier
  and therefore require a separate coordinated pruning protocol.
- Loro mergeable children are recent: introduced in 1.13.0, with an
  out-of-order import fix in 1.13.3 and deadlock/undo fixes in 1.13.6. The reverse
  replay fixture is a required regression gate before upgrading Loro.
- Protocol #33 must freeze revision IDs, envelope validation, version
  negotiation, and checkpoint compatibility before any fixture becomes a stable
  wire contract.
